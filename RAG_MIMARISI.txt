  1. GENEL BAKIS
================================================================================

  RAG sistemi, kullanicinin sorularina diyabet bilgi tabanindan ilgili
  referanslari bulup, bu referanslarla zenginlestirilmis yanitlar uretir.

  Hibrit erisim motoru: Semantik (FAISS) + Anahtar kelime (BM25) aramasi
  Capraz dil destegi: Turkce soru -> Ingilizce bilgi tabani eslemesi

================================================================================
  4. METIN PARCALAMA (Chunker)
================================================================================

  Dosya: rag/chunker.py

  4.1. Parametreler
  ------------------
  CHUNK_SIZE    = 1024 karakter (maksimum parca boyutu)
  CHUNK_OVERLAP = 128 karakter  (parcalar arasi ortusme)

  4.2. Recursive Splitting Algoritmasi
  -------------------------------------
  Oncelik sirali ayiricilar kullanilir:

  Oncelik   Ayirici     Aciklama
  ----------------------------------------
  1         "\n\n"      Paragraf sinirlari
  2         "\n"        Satir sinirlari
  3         ". "        Cumle sinirlari
  4         " "         Kelime sinirlari (fallback)

  Algoritma:
  1. Metin uzunlugu <= CHUNK_SIZE ise -> direkt dondur
  2. En yuksek oncelikli ayirici ile bol
  3. Greedy birlestirme:
     - Her parca icin:
       * aday = mevcut_parca + ayirici + yeni_parca
       * len(aday) <= CHUNK_SIZE ise -> birlestir
       * degilse:
         - Mevcut parcayi kaydet
         - Ortusme bolgesini al: son 128 karakter
         - Yeni parcayi baslat: ortusme + ayirici + yeni_parca
  4. Hicbir ayirici bulunamazsa: Sert karakter bolmesi
     - Her 1024 karakterde kes, 128 karakter geri git

  4.3. Metadata On-eki
  ---------------------
  Her parca kaynak bilgisiyle on-eklenir:
    "[Kaynak Basligi] parca metni..."

  Amac: Gomme modelinin farkli bilgi kaynaklarini ayirt etmesi

  4.4. Cikti Yapisi
  ------------------
  {
    "chunk_id":     int,    // Benzersiz parca kimlik numarasi
    "text":         str,    // "[kaynak] parca metni..."
    "source":       str,    // Kaynak basligi
    "page_number":  int     // Sayfa numarasi
  }

  4.5. Mevcut Istatistikler
  --------------------------
  Toplam parca sayisi: 698 chunk
  Ortalama parca boyutu: ~900 karakter


================================================================================
  5. GOMME VEKTORLERI (Embeddings)
================================================================================

  Dosya: rag/embeddings.py

  5.1. Model Bilgileri
  ---------------------
  Model:     intfloat/multilingual-e5-base
  Boyut:     768 boyutlu vektorler
  Dil:       100+ dil destegi (Turkce + Ingilizce dahil)
  Ozellik:   Asimetrik arama icin optimize (query vs passage on-eki)

  5.2. Dual Backend Mimarisi (Singleton Pattern)
  -----------------------------------------------
  Sistem, calisma ortamina gore otomatik backend secer:

  +---------------------+     +---------------------+
  | LOCAL BACKEND        |     | CONTAINER BACKEND   |
  | (Indeks Olusturma)   |     | (Sorgu Cevaplama)   |
  +---------------------+     +---------------------+
  | sentence-transformers|     | onnxruntime         |
  | ~1.1 GB              |     | + tokenizers        |
  | GPU/CPU              |     | Sadece CPU           |
  | embed_texts()        |     | embed_query()       |
  | Toplu gomme          |     | Tekil gomme          |
  +---------------------+     +---------------------+

  Backend secimi (otomatik):
    try:
      from sentence_transformers import SentenceTransformer
      BACKEND = "sentence-transformers"
    except ImportError:
      import onnxruntime
      BACKEND = "onnx"

  5.3. On-ek Kurali (E5 Modeli Gereksinimi)
  ------------------------------------------
  Metin Tipi    On-ek          Kullanim
  -------------------------------------------------
  Sorgu         "query: "      embed_query() icinde
  Dokuman       "passage: "    embed_texts() icinde

  Not: E5 modeli, asimetrik arama icin bu on-ekleri gerektirir.
       On-eksiz kullanim performansi dusurur.

  5.4. ONNX Inference Pipeline
  -----------------------------
  Adim 1: Tokenize
    - Tokenizer: onnx_model/tokenizer.json
    - pad_id = 1, pad_token = "<pad>"
    - max_length = 512 token
    - Padding ve truncation aktif

  Adim 2: Model Cikarimi
    - Girdi: input_ids (1, seq_len), attention_mask (1, seq_len)
    - Cikti: last_hidden_state (1, seq_len, 768)
    - Provider: CPUExecutionProvider

  Adim 3: Mean Pooling
    - token_embeddings = outputs[0]           // (1, seq_len, 768)
    - mask = attention_mask[:,:,np.newaxis]    // (1, seq_len, 1)
    - pooled = SUM(embeddings * mask) / SUM(mask)
    - Sonuc: (1, 768)

  Adim 4: L2 Normalizasyon
    - norm = ||pooled||_2
    - normalized = pooled / norm
    - Sonuc: Birim vektoru (1, 768), dtype=float32

  5.5. Model Dosyalari
  ---------------------
  Dosya                  Boyut        Aciklama
  ----------------------------------------------------------
  onnx_model/model.onnx       1.4 MB  ONNX model yapisi
  onnx_model/model.onnx.data  1.1 GB  Model agirliklari
  onnx_model/tokenizer.json   17 MB   Hizli tokenizer
  onnx_model/tokenizer_config.json     Tokenizer ayarlari


================================================================================
  6. VEKTOR DEPOSU (FAISS Vector Store)
================================================================================

  Dosya: rag/vector_store.py

  6.1. Indeks Tipi
  -----------------
  FAISS IndexFlatIP (Inner Product / Ic Carpim)

  Neden Inner Product?
  - Vektorler L2-normalize edilmis oldugu icin:
    inner_product(a, b) = cosine_similarity(a, b)
  - Tam arama (brute-force), yaklasik degil -> %100 recall
  - 698 vektor icin yeterli performans

  6.2. Depolama Yapisi
  ---------------------
  Dosya               Boyut     Icerik
  ----------------------------------------------------------
  faiss_index/index.faiss   2.1 MB   FAISS binary indeksi
  faiss_index/chunks.json   678 KB   Chunk metadata (JSON)

  chunks.json formati:
  [
    {
      "chunk_id": 0,
      "text": "[ADA Standards] HbA1c >= 6.5%...",
      "source": "ADA Standards of Care 2024",
      "page_number": 1
    },
    ...
  ]

  6.3. Arama Algoritmasi
  -----------------------
  search(query_embedding, top_k=5):
    1. Sorgu vektoru: (1, 768) float32
    2. FAISS arama: index.search(query_embedding, top_k)
    3. Donus: [(chunk_dict, similarity_score), ...]
    4. Gecersiz indeksler filtrelenir (idx < 0)

  6.4. Mevcut Istatistikler
  --------------------------
  Toplam vektor: 698
  Boyut: 768
  Indeks tipi: Tam arama (Flat)
  Arama suresi: ~1-2ms (698 vektor icin)


================================================================================
  7. HIBRIT ERISIM MOTORU (Hybrid Retriever)
================================================================================

  Dosya: rag/retriever.py

  3 asamali hibrit erisim: Semantik + Anahtar Kelime + RRF Fuzyon

  7.1. Asama 1: Semantik Arama (FAISS)
  --------------------------------------
  - Aday sayisi: fetch_k = min(top_k * 6, 60)
  - Sorgu gomu: embed_query(query) -> (1, 768)
  - FAISS arama: vector_store.search(embedding, fetch_k)
  - Donus: [(chunk, cosine_similarity_score), ...]
  - Agirlik: SEMANTIC_WEIGHT = 0.55

  Guc: Anlamsal benzerlik yakalar
       "seker hastaligi" <-> "diabetes mellitus" eslesmesi

  7.2. Asama 2: Anahtar Kelime Aramasi (BM25-lite)
  -------------------------------------------------
  BM25 Parametreleri:
    k1     = 1.5    (Terim frekansi doygunluk noktasi)
    b      = 0.75   (Uzunluk normalizasyon faktoru)
    avg_dl = otomatik hesaplanir (ortalama dokuman uzunlugu)

  BM25 Formulu:
    skor = SUM_i [ IDF(q_i) * TF_norm(q_i) ]

    IDF(q_i) = log( (N - df + 0.5) / (df + 0.5) + 1.0 )
      N  = toplam dokuman sayisi (698)
      df = terimi iceren dokuman sayisi

    TF_norm(q_i) = (tf * (k1 + 1)) / (tf + k1 * (1 - b + b * dl / avg_dl))
      tf     = terimin dokumandaki frekansi
      dl     = dokuman uzunlugu (token sayisi)
      avg_dl = ortalama dokuman uzunlugu

  Onisleme:
    1. Turkce karakter normalizasyonu: cgiosu -> cgiosu
    2. Kucuk harfe cevirme
    3. Token cikarimi: regex [a-z0-9]+
    4. Filtreleme: uzunluk > 2 olan tokenlar
    5. Capraz dil genisletme (Bolum 8'e bakiniz)

  Agirlik: KEYWORD_WEIGHT = 0.45

  Guc: Tam kelime eslesmesi yakalar
       "HbA1c" gibi tibbi terimler icin etkili

  7.3. Asama 3: RRF Fuzyon (Reciprocal Rank Fusion)
  --------------------------------------------------
  Iki arama sonucunu tek bir siraya birlestiren algoritma.

  RRF Parametresi:
    rrf_k = 60 (yumusatma sabiti)

  Formul:
    Semantik sonuclar icin (rank 0'dan baslayarak):
      rrf_skor += SEMANTIC_WEIGHT / (rrf_k + rank + 1)
      rrf_skor += 0.55 / (60 + rank + 1)

    BM25 sonuclar icin:
      base_rrf = KEYWORD_WEIGHT / (rrf_k + rank + 1)
      base_rrf = 0.45 / (60 + rank + 1)

      Eger chunk SADECE BM25'te bulunduysa (semantikte yok):
        base_rrf *= 2.5   (capraz dil esleme bonusu)

      rrf_skor += base_rrf

  RRF Ornegi:
    Chunk A: Semantik rank=0, BM25 rank=2
      rrf = 0.55/(60+1) + 0.45/(60+3) = 0.00902 + 0.00714 = 0.01616

    Chunk B: Sadece BM25 rank=0 (capraz dil esleme)
      rrf = 0.45/(60+1) * 2.5 = 0.01844

  7.4. Kaynak Cesitliligi
  ------------------------
  Siradan sonra: Her kaynaktan maksimum 2 chunk secilir.
  Amac: Farkli kaynaklardan bilgi sunarak cevabin kapsamini genisletmek.

  7.5. Minimum Skor Esigi
  -------------------------
  MIN_SCORE_THRESHOLD = 0.45
  Bu esik altindaki sonuclar filtrelenir.

  7.6. Final Cikti
  -----------------
  top_k = 5 sonuc (varsayilan)

  Her sonuc:
  {
    "text":   str,     // Chunk metni
    "source": str,     // Kaynak basligi
    "score":  float    // RRF skoru (yuvarlanmis, 4 basamak)
  }

  8. TURKCE-INGILIZCE CAPRAZ DIL DESTEGI

  Dosya: rag/retriever.py

  8.1. Amac
  ----------
  Bilgi tabani agirlikli olarak Ingilizce (91/101 dokuman).
  Kullanici Turkce soru sorar -> BM25 anahtar kelime aramasi
  Ingilizce dokumanlarda esleme bulamaz.

  Cozum: Turkce sorgu tokenlarini Ingilizce esdegerleriyle genisletme.

  8.2. Turkce Karakter Normalizasyonu
  ------------------------------------
  _normalize_turkish(text):
    c -> c,  g -> g,  i -> i,  o -> o,  s -> s,  u -> u
    C -> C,  G -> G,  I -> I,  O -> O,  S -> S,  U -> U

  Ornek: "seker hastaligi" -> "seker hastaligi" (zaten ASCII)
         "bobregin" -> "bobregin"

  8.4. Sorgu Genisletme Algoritmasi
  ----------------------------------
  _expand_query_tokens(tokens):

  1. Her sorgu tokeni icin:
  2.   Her Turkce anahtar icin:
  3.     Esleme uzunlugu = min(len(token), len(anahtar), 4)
  4.     Eger esleme >= 3 karakter:
  5.       token[:esleme] == anahtar[:esleme] ise:
  6.         Tum Ingilizce sinonimleri genisletilmis listeye ekle

  Ornek:
    Sorgu: "diyabette beslenme nasil olmali"
    Tokenlar: ["diyabette", "beslenme", "nasil", "olmali"]

    "beslenme" -> kok "bes" eslesmesi:
      -> ["nutrition", "diet", "dietary", "food", "eating"] eklenir

    Genisletilmis tokenlar:
      ["diyabette", "beslenme", "nasil", "olmali",
       "nutrition", "diet", "dietary", "food", "eating"]

  9.2. Gemini System Prompt'una Enjeksiyon
  -----------------------------------------
  BuildFullSystemPrompt(ragContext):

    System prompt sonu:
    """
    BILGI TABANI REFERANSLARI:
    Asagida kullancinin sorusuyla dogrudan ilgili bilgi tabanindan
    alinmis referans bilgiler var.

    KULLANIM KURALLARI:
    1. Yanitini ONCELIKLE bu referans bilgilere dayandir
    2. Genel bilgi yerine referanslardaki SOMUT bilgiyi kullan
       (orn: "ADA/TEMD kilavuzuna gore...")
    3. Bilgiyi hangi kaynaktan aldigini parantez icinde belirt
       (Kaynak: ...)
    4. Referanslarda BULUNMAYAN bir bilgi veriyorsan, bunu kendi
       genel bilgin olarak belirt
    5. Tum referanslari listeleme, sadece soruyla en ilgili

  11. INDEKS OLUSTURMA
================================================================================

  Dosya: scripts/build_index.py

  4 Asamali Pipeline:

  Asama 1: Dokuman Yukleme
    load_documents(KNOWLEDGE_BASE_DIR)
    -> 101 dosyadan dokuman sayfalari yuklenir

  Asama 2: Metin Parcalama

    -> 698 chunk olusturulur (1024 karakter, 128 ortusme)

  Asama 3: Gomme Vektoru Uretimi
    embedding_service.embed_texts(texts)
    -> 698 adet (768,) boyutlu vektor uretilir

    vector_store.build_index(chunks, embeddings)
    vector_store.save()
    -> faiss_index/index.faiss (2.1 MB)
    -> faiss_index/chunks.json (678 KB)

  Not: Bu islem LOCAL ortamda calistirilir (sentence-transformers gerekli).
       Container icinde ONNX backend kullanildigi icin indeks olusturulamaz.
